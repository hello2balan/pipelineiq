Style fingerprint
How does this engineer structure DataFrames? Handle nulls? Reference Unity Catalog tables? Extract it all automatically.
7 dimensions captured
DataFrame style
Method chaining vs intermediate assignments vs spark.sql() preference
Null handling
Defensive checks before joins, coalesce patterns, or trust upstream
UC table references
Fully qualified catalog.schema.table vs USE statement vs short names
Entity filter approach
How LH, LX, OS, SN are filtered -- explicit list, wildcard, or dynamic config
Error handling
try/except depth, audit log writes, retry logic or none
Date logic
Departure date vs booking date vs event time watermarks
FX rate approach
lhg_prod.ref.fx_rates with join date vs hardcoded rate vs custom model

Risk register
What is fragile? What assumptions are hardcoded? What would break if touched? Flagged automatically before it becomes a problem.
4 risk categories detected
Institutional knowledge risk
Years of implicit context embedded in code -- not documented anywhere else
Example: FX logic with finance sign-off from Oct 2023 -- undocumented
Hardcoded values
Dates, rates, thresholds baked into code that should be config-driven
Example: EUR rate hardcoded at 1.08 instead of lhg_prod.ref.fx_rates
Silent failure points
Joins without null checks that will silently corrupt data on schema changes
Example: Passenger join with no null validation on passenger_id
Single point of failure
Technology or pattern only this engineer uses -- no team backup exists
Example: Delta Live Tables -- no other engineer on the team uses DLT

LHG coding standard -- 7 rules
1
UC table references must always be fully qualified (catalog.schema.table)
2
Null validation must be performed before every join operation
3
Entity filters must explicitly list all applicable codes -- no wildcard
4
All notebooks must include try/except blocks with writes to the audit log table
5
Schema validation must run at notebook start before any transformation
6
Departure date must be used as the primary date key for booking pipelines
7
EUR FX conversion must use lhg_prod.ref.fx_rates -- no hardcoded rates

About PipelineIQ
AI-powered code intelligence for Databricks data engineering teams
What PipelineIQ solves
When a data engineer leaves, they take years of implicit knowledge with them. Why the entity filter excludes certain codes. Which FX rate table to use and why. Why that join has a 2-hour delay built in.

PipelineIQ extracts that knowledge automatically from the code itself, before the engineer walks out the door.
What it produces
Style fingerprint -- how this engineer codes vs the team standard

LHG knowledge map -- entity logic, date conventions, FX approach

Risk register -- what is fragile, what only this person understands

Handover brief -- clear language, 30 minutes to read
Privacy and tokenisation
Before any notebook is analysed, a tokenisation layer runs entirely in your browser:
Before (in your browser)
host="lhg-prod-db.internal" password="Lh@2024Prod!" table="lhg_prod.booking.fact_v3"
After (sent to Claude)
host="HOST_TOKEN_001" password="CRED_TOKEN_002" table="TABLE_TOKEN_003"
The token map stays in your browser only. Claude sees the logic and structure -- never the actual credentials or internal identifiers.
